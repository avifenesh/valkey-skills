## Fixes Applied

1. **deploy.sh deadlock** - Replaced `kubectl rollout status` (waits for Ready) with per-pod `kubectl wait --for=jsonpath='{.status.phase}'=Running`. The readiness probe checks `cluster_state:ok`, which can't pass until the init job runs after the wait - a chicken-and-egg deadlock.

2. **generate-tls.sh SAN bug** - `DNS.3` expanded to `valkey-client.valkey-cluster.valkey-cluster.svc.cluster.local` (doubled namespace). Fixed by using `${NAMESPACE}` variable directly for the client service SAN.

3. **DEPLOYMENT_GUIDE.md** - Rewrote to match actual manifests. The previous version referenced a non-existent `valkey-lb` LoadBalancer service, had the same rollout status deadlock in manual steps, mentioned init containers that don't exist, and stated 20Gi storage when the StatefulSet uses 10Gi.

## Key Design Decisions

- **`valkey/valkey-bundle:9`** image - includes valkey-search pre-loaded; the ConfigMap also explicitly loads it via `loadmodule` for clarity
- **Readiness probe checks `cluster_state:ok`** rather than just PING - this removes nodes from the client service when they lose cluster membership, which is the right production behavior. The tradeoff is that initial deployment must wait for Running phase (not Ready) before running the init job
- **`podManagementPolicy: Parallel`** - all 6 pods start simultaneously instead of sequentially, which speeds up initial deployment
- **`publishNotReadyAddresses: true`** on the headless service - ensures DNS records exist even when pods are not Ready, which is required for the init job to reach pods before cluster formation
- **`tls-auth-clients no`** - server-only TLS (no mutual TLS by default); the guide recommends enabling mTLS for production
- **No CPU limits** - avoids CFS throttling latency spikes per the skill reference
