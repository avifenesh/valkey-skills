## Root Cause Analysis: Dual-Primary Split-Brain After Partition Heal

### The Symptom

After a network partition heals, two nodes both claim primary ownership of the same slots. `CLUSTER INFO` shows `cluster_state:ok` but `CLUSTER NODES` shows overlapping slot ownership. The `currentEpoch` on both conflicting nodes is the same value - the epoch did not advance during the failover.

---

### The Normal Failover Epoch Path

In a healthy failover, the epoch advances at exactly one point. Here is the code path:

**File: `cluster_legacy.c:5766-5773`** - inside `clusterHandleReplicaFailover()`:
```c
/* Ask for votes if needed. */
if (server.cluster->failover_auth_sent == 0) {
    server.cluster->currentEpoch++;                              // <-- THE epoch increment
    server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
    clusterRequestFailoverAuth();
    server.cluster->failover_auth_sent = 1;
    clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG | CLUSTER_TODO_UPDATE_STATE | CLUSTER_TODO_FSYNC_CONFIG);
    return; /* Wait for replies. */
}
```

This is the only place `currentEpoch` gets incremented during automatic failover. The replica bumps its local `currentEpoch`, stores it as `failover_auth_epoch`, then broadcasts a `FAILOVER_AUTH_REQUEST` to all nodes.

Other nodes learn the new epoch at **`cluster_legacy.c:3898-3903`** when they receive the message:
```c
if (sender && !nodeInHandshake(sender)) {
    sender_claimed_current_epoch = ntohu64(msg->currentEpoch);
    if (sender_claimed_current_epoch > server.cluster->currentEpoch)
        server.cluster->currentEpoch = sender_claimed_current_epoch;
}
```

If the request reaches a majority of voting primaries, they grant votes via `clusterSendFailoverAuthIfNeeded()` (**line 5268**). The winning replica then sets its `configEpoch` to the election epoch (**line 5784**) and calls `clusterFailoverReplaceYourPrimary()` to claim the slots.

---

### The Six Conditions That Must Be Met for a Vote

Each voting primary checks these gates in `clusterSendFailoverAuthIfNeeded()` (**lines 5268-5364**):

| # | Condition | Code Line | What It Checks |
|---|-----------|-----------|----------------|
| 1 | `clusterNodeIsVotingPrimary(myself)` | 5281 | Voter must be a primary with >= 1 slot |
| 2 | `server.cluster->safe_to_join` | 5283 | Not in startup delay (first 2s after boot while in CLUSTER_FAIL state) |
| 3 | `requestCurrentEpoch >= server.cluster->currentEpoch` | 5293 | Request epoch not stale |
| 4 | `lastVoteEpoch != currentEpoch` | 5301 | Haven't already voted in this epoch |
| 5 | `!nodeFailed(primary) && !force_ack` | 5310 | Requester's primary must be in FAIL state (or FORCEACK set) |
| 6 | `slot_owner->configEpoch <= requestConfigEpoch` | 5333 | No current slot owner has a higher configEpoch for claimed slots |

**Condition 5 is the critical one for this scenario.**

### Root Cause: Asymmetric Partition Prevents FAIL Quorum on Voting Majority

Here is the scenario that produces the observed state:

**Setup**: Nodes A (primary, slots 0-5460), B (primary, slots 5461-10922), C (primary, slots 10923-16383), with replicas A', B', C'.

**Partition topology**: The network splits such that:
- **Partition 1**: A, B, C' (the majority of primaries)
- **Partition 2**: A', B', C

This is an asymmetric partition. From Partition 1's perspective:

1. C is unreachable. A and B mark C as PFAIL. With 2 out of 3 voting primaries reporting PFAIL, quorum is met (`(3/2)+1 = 2`). C is promoted to FAIL via `markNodeAsFailingIfNeeded()` (**line 5268** voter check requires `nodeFailed(primary)` at line 5310).

2. C' (replica of C, in Partition 1) sees its primary C marked FAIL. `clusterHandleReplicaFailover()` fires (**line 5616-5618**):
   ```c
   if (clusterNodeIsPrimary(myself) || myself->replicaof == NULL ||
       (!nodeFailed(myself->replicaof) && !manual_failover) || ...)
   ```
   C' proceeds because `nodeFailed(myself->replicaof)` is true.

3. C' increments `currentEpoch` (**line 5767**), broadcasts `FAILOVER_AUTH_REQUEST` to all reachable nodes (A and B).

4. A and B check condition 5: is C (the old primary) in FAIL state? **Yes**, they marked it FAIL. They grant votes. C' wins with 2 votes (A + B), quorum is 2. C' becomes the new primary for slots 10923-16383.

**But from Partition 2's perspective**:

5. A and B are unreachable. C cannot reach a quorum to mark A or B as FAIL (C alone is 1 vote, quorum requires 2). **No FAIL promotion happens** in Partition 2.

6. C remains a primary serving its slots. It never sees the epoch increment from C' because C' is in the other partition. C's `currentEpoch` stays at its pre-partition value.

7. A' and B' see their primaries as unreachable (PFAIL), but with only C as a voting primary in this partition, there's only 1 out of 3 votes - below quorum. A' and B' cannot get their primaries marked FAIL either. **No failovers happen in Partition 2.**

**When the partition heals**:

8. C reconnects to the cluster. It broadcasts PINGs with its old `configEpoch` and its old slot bitmap claiming slots 10923-16383.

9. C' also broadcasts PINGs with its new (higher) `configEpoch` for the same slots.

10. Here is the critical problem: **the epoch update propagation at line 3902 only updates `currentEpoch`**, not `configEpoch`. When C receives a message from C' or any node in the healed cluster, C's `currentEpoch` catches up:
    ```c
    if (sender_claimed_current_epoch > server.cluster->currentEpoch)
        server.cluster->currentEpoch = sender_claimed_current_epoch;
    ```
    This makes the `currentEpoch` values converge to the same number on both nodes. **But C's `configEpoch` for its own node entry is not changed by this** - only the global cluster-level `currentEpoch` is updated.

11. The slot conflict resolution at **line 3078-3080** should resolve this:
    ```c
    if (isSlotUnclaimed(j) ||
        server.cluster->slots[j]->configEpoch < senderConfigEpoch ||
        clusterSlotFailoverGranted(j))
    ```
    C' has a higher `configEpoch` than C for those slots, so nodes receiving C''s PING will rebind the slots to C'. **But C itself needs to receive this update.**

12. At **lines 4317-4351**, when a node sees a sender claiming slots that belong to a node with a higher `configEpoch`, it sends an UPDATE message back:
    ```c
    if (slot_owner->configEpoch > sender_claimed_config_epoch) {
        clusterSendUpdate(sender->link, slot_owner);
    }
    ```
    This should inform C that its slots now belong to C'. When C processes the UPDATE at **line 4446**:
    ```c
    clusterUpdateSlotsConfigWith(n, reportedConfigEpoch, msg->data.update.nodecfg.slots);
    ```
    C should give up the slots.

**But here is where the race lives.** Between steps 10-12, there is a window where:
- Both C and C' are serving the same slots
- `cluster_state` is `ok` on both (because `clusterUpdateState()` at **line 6604** sees all slots covered)
- The `currentEpoch` has converged (identical values on both)
- But the UPDATE message hasn't been received/processed yet

---

### Why `currentEpoch` Appears Identical

The epoch **did** advance during failover - but only in Partition 1 (at line 5767). When C in Partition 2 reconnects, line 3902 immediately pulls its `currentEpoch` up to match. This makes the `currentEpoch` on both conflicting nodes identical despite only one having participated in the election.

The `currentEpoch` being the same is actually the **expected** converged state. The issue isn't that the epoch failed to increment - it's that the **configEpoch** difference hasn't yet propagated to force C to step down.

---

### What Delays Resolution

Three factors can extend the window where both nodes claim the same slots:

**1. UPDATE message delivery is one-slot-at-a-time** (**line 4355**):
```c
/* TODO: instead of exiting the loop send every other
 * UPDATE packet for other nodes that are the new owner
 * of sender's slots. */
found_new_owner = true;
break;
```
The code sends one UPDATE and breaks. If C owns 5461 slots, it takes many gossip rounds for C to learn about all slot reassignments. Each PING/PONG exchange corrects at most one slot range.

**2. FAIL flag cleared too early** - `clearNodeFailureIfNeeded()` at **line 2629-2630**:
```c
if (clusterNodeIsVotingPrimary(node) &&
    (now - node->fail_time) > (server.cluster_node_timeout * CLUSTER_FAIL_UNDO_TIME_MULT))
```
If `cluster_node_timeout * 2` has elapsed since C was marked FAIL, C's FAIL flag is cleared when it reconnects. C is now a voting primary again and could potentially participate in new elections, complicating recovery.

**3. `clusterHandleConfigEpochCollision()`** at **line 2449** only fires when two primaries have the **same** configEpoch. If C kept configEpoch=X and C' got configEpoch=X+1, there's no collision to resolve. The update path is purely gossip-driven via UPDATE messages.

---

### The Specific Condition That Was Not Met

No condition "failed" in the code. The epoch increment at line 5767 happened correctly. The failover election succeeded correctly. The root cause is a **convergence delay**, not a logic failure:

1. The partition prevented C from receiving the epoch increment in real time
2. After partition heal, `currentEpoch` converges instantly (line 3902) but slot ownership (`configEpoch` comparison) converges through gossip UPDATE messages
3. The UPDATE loop exits after one slot range per exchange (line 4355), serializing the convergence
4. During this convergence window, both nodes serve the same slots

### Remediation

**Immediate**: Run `CLUSTER FAILOVER` on the correct node (C') if it has the more current data. Or simply wait - the gossip protocol will converge, though it may take `O(num_slots / gossip_rate)` time.

**Monitoring**: Alert on `CLUSTER NODES` showing two primaries in the same shard. The `cluster_state:ok` from `CLUSTER INFO` is misleading during this convergence window because all slots are "covered" (just by two different nodes).

**Code-level improvement**: The TODO at line 4352-4354 acknowledges the single-UPDATE-per-PING limitation. Sending UPDATE messages for all conflicting slot owners in a single pass would dramatically reduce the convergence window.
