## Root Cause Analysis: Split-Brain Slot Ownership After Partition Heal

### 1. The Epoch Increment Code Path

The `currentEpoch` is incremented in exactly one place during automatic failover: inside `clusterHandleReplicaFailover()` in `cluster_legacy.c`, at the moment the replica broadcasts its vote request (Step 2 of the election):

```c
// cluster_legacy.c - clusterHandleReplicaFailover()
server.cluster->currentEpoch++;
server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
clusterRequestFailoverAuth();          // broadcast FAILOVER_AUTH_REQUEST
server.cluster->failover_auth_sent = 1;
```

This happens **locally on the replica** that initiates the election. The incremented epoch is then carried in the `FAILOVER_AUTH_REQUEST` message header to all nodes. Other nodes adopt it via the epoch-sync path in `clusterProcessPacket()`:

```c
// cluster_legacy.c - clusterProcessPacket()
if (senderCurrentEpoch > server.cluster->currentEpoch)
    server.cluster->currentEpoch = senderCurrentEpoch;
```

If the replica never reaches this code, the epoch never advances anywhere in the cluster.

---

### 2. Preconditions That Must Be Met Before the Increment

`clusterHandleReplicaFailover()` has a series of early-return guards. Every one of these must pass before the epoch increment line is reached:

**Guard 1 - Must be a replica:**
```c
if (nodeIsPrimary(myself)) return;
```

**Guard 2 - Must have a primary assigned:**
```c
if (myself->replicaof == NULL) return;
```

**Guard 3 - Primary must be in FAIL state (or manual failover active):**
```c
if (!nodeFailed(myself->replicaof) && !manual_failover) return;
```

**Guard 4 - `cluster-replica-no-failover` must not be set:**
```c
if (server.cluster_replica_no_failover && !manual_failover) return;
```

**Guard 5 - Replica data must be recent enough:**
```c
if (data_age > server.cluster_node_timeout *
    server.cluster_replica_validity_factor + repl_ping_period)
    return;  // Data too stale, refuse to failover
```

**Guard 6 - Vote request not already sent and scheduled time reached:**
```c
if (server.cluster->failover_auth_sent == 0 &&
    mstime() >= server.cluster->failover_auth_time) {
    // Only now: increment epoch and send vote request
}
```

Guard 3 is the critical one for this scenario. The primary must be in the FAIL state (not merely PFAIL) before the replica will request votes.

---

### 3. The FAIL Quorum Gate (The Most Likely Blocker)

PFAIL is promoted to FAIL in `markNodeAsFailingIfNeeded()`:

```c
// cluster_legacy.c - markNodeAsFailingIfNeeded()
int needed_quorum = (server.cluster->size / 2) + 1;

failures = clusterNodeFailureReportsCount(node);
if (clusterNodeIsVotingPrimary(myself)) failures++;  // Count self
if (failures < needed_quorum) return;                 // NOT ENOUGH - no FAIL
```

`server.cluster->size` is the number of primaries holding at least one slot. In a 3-primary cluster, `size = 3`, so `needed_quorum = 2`.

**Failure reports can only come from voting primaries** (primaries with slots). Replicas and slot-less primaries do not count. Reports also expire after `cluster_node_timeout * CLUSTER_FAIL_REPORT_VALIDITY_MULT` (2x the timeout).

For the partitioned side containing the replica to declare FAIL, it needs **2 voting primaries** (including potentially itself, but the replica isn't a voting primary) to report PFAIL for the old primary. This means at least 2 of the 3 primaries must be on the same side as the replica and must all independently detect the old primary as PFAIL.

---

### 4. What Prevented the Epoch From Advancing

Given a 6-node cluster with 3 primaries (P1, P2, P3) and 3 replicas (R1, R2, R3), here is the partition scenario that produces the observed symptoms:

```
Partition A:  P1, R2, R3          Partition B:  P2, P3, R1
(1 voting primary)                (2 voting primaries)
```

**On Partition B (P2, P3, R1):**
- R1 detects P1 is unreachable (PFAIL after `cluster_node_timeout`)
- P2 detects P1 is unreachable (PFAIL)
- P3 detects P1 is unreachable (PFAIL)
- P2 receives P3's PFAIL report via gossip: `failures = 2` (P3's report + self) >= `quorum = 2`
- P2 calls `markNodeAsFailing()`, broadcasts FAIL for P1
- R1 sees `nodeFailed(myself->replicaof)` is true - Guard 3 passes
- R1 enters `clusterHandleReplicaFailover()`, increments `currentEpoch` from N to N+1
- R1 wins the election (P2 and P3 vote, 2/3 = majority), sets `configEpoch = N+1`
- R1 calls `clusterFailoverReplaceYourPrimary()`, claims P1's slots

**On Partition A (P1, R2, R3):**
- P1 is the only voting primary
- R2 and R3 detect P2 and P3 as PFAIL, but they are replicas - they can't file failure reports
- P1 detects P2 and P3 as PFAIL, but alone it has only 1 failure report per node (itself)
- `failures = 1 < quorum = 2` - FAIL is **never declared** for P2 or P3
- R2 and R3 never enter the election path - **their Guard 3 blocks**
- P1 continues operating, still claims its original slots with `configEpoch = M` (where M < N+1)

**After partition heals:**
- Gossip flows between both sides
- `clusterProcessPacket()` syncs `currentEpoch` to the max: all nodes converge to N+1
- **Both P1 (old) and R1 (new primary) now show `currentEpoch = N+1`**

This is why the user observes the same `currentEpoch` on both conflicting nodes. The epoch DID advance (on Partition B during R1's election), but after gossip sync, P1 adopted the same value. The user sees N+1 on both and interprets it as "the epoch did not advance" because P1 never went through an election.

---

### 5. Why Both Still Claim the Same Slots

Normally this resolves quickly. When P1 receives gossip from R1 showing R1 owns P1's old slots with `configEpoch = N+1`, and P1's `configEpoch = M` where `M < N+1`, the conflict resolution in `clusterProcessPacket()` should fire:

```c
// cluster_legacy.c - clusterProcessPacket(), processing sender's slot bitmap
if (server.cluster->slots[j] != sender &&
    sender_config_epoch > server.cluster->slots[j]->configEpoch) {
    // Send UPDATE message to current owner, or update locally
}
```

**Three conditions can prevent or delay this resolution:**

**Condition A - P1 has `cluster-allow-reads-when-down yes` or `cluster-allow-pubsubshard-when-down yes`:**
P1 on the minority side would have entered `cluster_state:fail` (minority partition check in `clusterUpdateState()`), but with these configs it kept serving. After partition heals, `clusterUpdateState()` runs again and may set `cluster_state:ok` before the slot conflict is fully resolved via gossip. The user runs `CLUSTER INFO` at this moment and sees `cluster_state:ok` with overlapping ownership.

**Condition B - configEpoch collision:**
If P1's `configEpoch` was bumped to the same value as R1's during or after partition (via `clusterHandleConfigEpochCollision()` or an unrelated `clusterBumpConfigEpochWithoutConsensus()` call), neither node has a strictly higher configEpoch:

```c
// cluster_legacy.c - clusterHandleConfigEpochCollision()
// Only bumps if sender has GREATER node name (lexicographic)
if (sender->configEpoch == myself->configEpoch &&
    memcmp(sender->name, myself->name, CLUSTER_NAMELEN) > 0) {
    server.cluster->currentEpoch++;
    myself->configEpoch = server.cluster->currentEpoch;
}
```

If the collision detection function fires for the wrong pair of nodes, or if both sides bump to the same value through independent paths, the slot conflict persists because neither configEpoch is strictly greater.

**Condition C - Gossip propagation delay:**
UPDATE messages propagate slot ownership changes, but gossip is probabilistic. `clusterCron()` pings one random node per invocation (~10 Hz). It takes multiple rounds of gossip for all nodes to converge. During this window, `CLUSTER NODES` shows overlapping ownership on nodes that haven't yet processed the UPDATE.

---

### 6. The Most Likely Root Cause

**The epoch DID advance** - but only on the partition with the majority. P1 (old primary) was on the minority side and never saw the increment until the partition healed. After healing, gossip synced `currentEpoch` to the same value on both sides, creating the appearance that "the epoch did not advance."

The persistent split-brain is caused by one of:

1. **Timing window** (transient): The user observed `CLUSTER NODES` during the gossip convergence period after partition heal, before the UPDATE message reached P1. Running `CLUSTER NODES` again after a few seconds of gossip should show resolution.

2. **configEpoch collision** (persistent): If P1's `configEpoch` equals R1's `configEpoch`, `clusterHandleConfigEpochCollision()` in `cluster_legacy.c` must fire to break the tie. This requires both nodes to exchange a PING/PONG where they see the collision. If the collision handler's lexicographic comparison (`memcmp(sender->name, myself->name)`) doesn't trigger a bump (because it only bumps one side), there may be a delay of one more gossip round.

3. **Stale cluster config file** (persistent): If P1 was restarted during the partition and loaded its `nodes.conf` from disk, it may have an old view of the cluster that doesn't reflect the failover. The loaded `configEpoch` from `nodes.conf` overrides whatever gossip would provide during boot.

---

### 7. Diagnostic Steps and Resolution

**Verify the actual configEpoch values** (not currentEpoch - that's cluster-wide):
```
CLUSTER NODES  # on both conflicting nodes
```
Look at column 7 (config-epoch) for each node. If they differ, gossip will resolve it within seconds. If they're the same, that's the configEpoch collision case.

**Force resolution:**
```
# On the node that SHOULD yield (the old primary):
CLUSTER RESET SOFT
# Then re-add it as a replica:
CLUSTER REPLICATE <new-primary-node-id>
```

**Prevent recurrence:**
- Ensure `cluster-require-full-coverage yes` (default) so minority partitions stop serving
- Verify `cluster-replica-validity-factor` is not 0 (0 means replicas always attempt failover regardless of data age)
- Monitor `cluster_current_epoch` and `cluster_my_epoch` in `CLUSTER INFO` across all nodes for divergence

The critical invariant: **the epoch increment on line `server.cluster->currentEpoch++` is gated by Guard 3 (`nodeFailed(myself->replicaof)`), which is gated by `markNodeAsFailingIfNeeded()` achieving quorum, which requires a majority of voting primaries on the same partition as the replica.** If the partition splits voting primaries such that neither side has a majority alongside a replica of the failed primary, no election runs, no epoch advances, and slot ownership remains with the original primary - leaving any partial topology state changes unresolved when the partition heals.
