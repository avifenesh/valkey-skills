## Root Cause Analysis: split-brain after network partition - epoch not advanced during failover

**Source file**: `src/cluster_legacy.c`

---

### Cluster epoch model

Two distinct epoch values are involved:

- `server.cluster->currentEpoch` - cluster-wide monotonic logical clock, equalized across all nodes via gossip
- `clusterNode->configEpoch` (per node) - configuration version used to resolve slot ownership conflicts; the node with the higher `configEpoch` wins when two nodes claim the same slot

Both values live in `clusterState` (`cluster_legacy.h`):

```c
struct clusterState {
    uint64_t currentEpoch;       // cluster-wide clock
    uint64_t failover_auth_epoch; // epoch used for this election
    int      failover_auth_sent;  // 1 = vote request already sent
    uint64_t lastVoteEpoch;       // last epoch in which we voted
    ...
};

struct _clusterNode {
    uint64_t configEpoch;        // this node's config version
    ...
};
```

### How the epoch normally advances during automatic failover

The failover state machine runs in `clusterHandleReplicaFailover()`, called from both `clusterCron()` and `clusterBeforeSleep()`.

The epoch increment is at `cluster_legacy.c:5766`:

```c
/* Ask for votes if needed. */
if (server.cluster->failover_auth_sent == 0) {
    server.cluster->currentEpoch++;
    server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
    serverLog(LL_NOTICE,
        "Starting a failover election for epoch %llu, node config epoch is %llu",
        (unsigned long long)server.cluster->currentEpoch,
        (unsigned long long)nodeEpoch(myself));
    clusterRequestFailoverAuth();
    server.cluster->failover_auth_sent = 1;
    clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG | CLUSTER_TODO_UPDATE_STATE
                        | CLUSTER_TODO_FSYNC_CONFIG);
    return; /* Wait for replies. */
}
```

Conditions that must ALL be true to reach this block:

1. The node is a replica with its primary in FAIL state
2. `failover_auth_time <= now` (rank-based delay has elapsed)
3. `auth_age <= auth_timeout` (the election hasn't expired)
4. `failover_auth_sent == 0` (first call in this election round)

If any of conditions 1-3 fail, the function returns early and the epoch is never incremented in this cycle. If condition 4 fails, the election is already past the increment point.

When a vote request is received by another primary, it grants a vote only if (checked in `clusterSendFailoverAuthIfNeeded()`):

- The request epoch >= its own `currentEpoch`
- It hasn't already voted in this epoch (`lastVoteEpoch != currentEpoch`)

This guarantees at most one winner per epoch. After a quorum (`cluster->size / 2 + 1`) of ACKs arrive, `clusterFailoverReplaceYourPrimary()` is called at `cluster_legacy.c:5533`, which claims the primary's slots and sets `myself->configEpoch = failover_auth_epoch`. Under normal automatic failover, the epoch advances and everything converges.

### The TAKEOVER path and the defective bump

`CLUSTER FAILOVER TAKEOVER` bypasses the vote-based election entirely. Its handler is at `cluster_legacy.c:8159`:

```c
if (takeover) {
    serverLog(LL_NOTICE,
        "Taking over the primary (user request from '%s').", client);
    clusterBumpConfigEpochWithoutConsensus();   /* return value discarded */
    clusterFailoverReplaceYourPrimary();
}
```

The critical function is `clusterBumpConfigEpochWithoutConsensus()` at `cluster_legacy.c:2389`:

```c
int clusterBumpConfigEpochWithoutConsensus(void) {
    uint64_t maxEpoch = clusterGetMaxEpoch();

    if (myself->configEpoch == 0 || myself->configEpoch != maxEpoch) {
        server.cluster->currentEpoch++;
        myself->configEpoch = server.cluster->currentEpoch;
        clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG
                            | CLUSTER_TODO_FSYNC_CONFIG
                            | CLUSTER_TODO_BROADCAST_ALL);
        serverLog(LL_NOTICE, "New configEpoch set to %llu",
                  (unsigned long long)myself->configEpoch);
        return C_OK;
    } else {
        return C_ERR;   /* <<< silent failure - no increment, no log */
    }
}
```

The function increments only when:

```
myself->configEpoch == 0   (never been promoted before)
OR
myself->configEpoch != maxEpoch   (already behind the cluster max)
```

**The condition evaluates to false** - and the bump is silently skipped - when:

```
myself->configEpoch != 0   AND   myself->configEpoch == maxEpoch
```

### What `clusterGetMaxEpoch()` does (cluster_legacy.c:2345)

```c
uint64_t clusterGetMaxEpoch(void) {
    uint64_t max = 0;
    dictIterator *di;
    dictEntry *de;

    di = dictGetSafeIterator(server.cluster->nodes);
    while ((de = dictNext(di)) != NULL) {
        clusterNode *node = dictGetVal(de);
        if (node->configEpoch > max) max = node->configEpoch;  /* includes FAIL nodes */
    }
    dictReleaseIterator(di);
    if (max < server.cluster->currentEpoch) max = server.cluster->currentEpoch;
    return max;
}
```

This iterates **every node in the node table regardless of PFAIL/FAIL state**. An isolated, FAIL-marked primary still contributes its last-known `configEpoch` to the max. This is the pivot point of the failure.

### The specific precondition in your cluster

The bump silently fails only when `myself->configEpoch != 0`. In a cluster that has never failed over, replicas have `configEpoch = 0` and this path is safe. The non-zero value is introduced by a prior promotion that was not followed by a restart.

The trigger sequence:

**Round 1 (pre-history, before the partition you observed)**

1. A prior network event caused R1 (replica of P1) to win an election. R1 was promoted with `configEpoch = N`.
2. P1 came back online. P1's `configEpoch` was also `N` (from its config file), or was advanced slightly higher by some means, but through re-convergence both ended at `N`.
3. R1 was demoted back to replica of P1, either via `CLUSTER REPLICATE` or automatically when `clusterUpdateSlotsConfigWith()` determined P1 should be primary.

**The critical gap**: `clusterSetPrimary()` at `cluster_legacy.c:6780` changes the node's flags (`PRIMARY → REPLICA`) and sets `replicaof`, but **does not reset `myself->configEpoch` to 0**:

```c
static void clusterSetPrimary(clusterNode *n, int closeSlots, int full_sync_required) {
    serverAssert(n != myself);
    serverAssert(myself->numslots == 0);

    if (clusterNodeIsPrimary(myself)) {
        myself->flags &= ~(CLUSTER_NODE_PRIMARY | CLUSTER_NODE_MIGRATE_TO);
        myself->flags |= CLUSTER_NODE_REPLICA;    /* role change only */
    }
    /* ... configEpoch is never touched ... */
    myself->replicaof = n;
    ...
}
```

The `nodes.conf` file does reset `configEpoch` to 0 for replicas on disk load (`cluster_legacy.c:968`):

```c
n->configEpoch = (nodeIsReplica(n) && n->replicaof) ? 0 : strtoull(argv[6], NULL, 10);
```

But this only runs at startup, not on runtime demotion. If R1 was demoted without a restart, it continues running with `myself->configEpoch = N`.

**Round 2 (the partition you observed)**

State entering the partition:
- P1: `configEpoch = N`, primary for slots 0-5461, `currentEpoch = N`
- R1: `configEpoch = N` (retained from prior promotion), currently a replica, `currentEpoch = N`
- P2, P3: `configEpoch < N`
- Cluster max epoch: `N`

P1 becomes isolated. Someone issues `CLUSTER FAILOVER TAKEOVER` to R1 (or it is issued automatically by an operator script because automatic failover couldn't achieve quorum in time).

Call chain:
```
CLUSTER FAILOVER TAKEOVER handler
  → clusterBumpConfigEpochWithoutConsensus()
      → maxEpoch = clusterGetMaxEpoch()
          → iterates all nodes including FAIL-marked P1
          → P1->configEpoch = N, returns N
      → condition: (N == 0 || N != N)
                    false     false
      → condition is false
      → returns C_ERR    <-- epoch NOT incremented, no log emitted
  → clusterFailoverReplaceYourPrimary()    <-- runs unconditionally
      → clusterSetNodeAsPrimary(myself)
      → claims P1's slots (clusterAddSlot, sets slots[j] = myself)
      → broadcasts PONG with configEpoch = N, slot bitmap updated
```

R1 is now primary for slots 0-5461 with `configEpoch = N` and `currentEpoch = N`. P1 is still running isolated with `configEpoch = N` and `currentEpoch = N`.

### Why partition heal doesn't auto-resolve

When the partition heals, P1 and R1 exchange PING/PONG. P1 receives R1's PONG claiming slots 0-5461 with `configEpoch = N`. `clusterUpdateSlotsConfigWith()` is called at `cluster_legacy.c:3019`. The slot transfer condition is:

```c
if (isSlotUnclaimed(j) ||
    server.cluster->slots[j]->configEpoch < senderConfigEpoch ||  /* strict < */
    clusterSlotFailoverGranted(j)) {
    /* transfer slot ownership to sender */
}
```

`server.cluster->slots[j]->configEpoch` (P1's view of itself) = `N`.
`senderConfigEpoch` (R1) = `N`.
Condition: `N < N` → **false**. P1 does not yield its slots to R1.

Symmetrically, R1 receives P1's PONG, evaluates the same condition, and does not yield either. Both nodes update their `currentEpoch` to `max(N, N) = N` via gossip - so `currentEpoch` is identical on both.

`CLUSTER NODES` shows both advertising the same slot range. `CLUSTER INFO` shows `cluster_state:ok` because all slots are covered by at least one non-FAIL node (`clusterUpdateState()` only cares that every slot has an owner, not that ownership is unique).

### The collision resolution mechanism and why it's slow

`clusterHandleConfigEpochCollision()` is the safety net (called from `clusterProcessPacket()` when a PING/PONG arrives from a primary, `cluster_legacy.c:4364`):

```c
if (sender && nodeIsPrimary(myself) && nodeIsPrimary(sender) &&
    sender_claimed_config_epoch == myself->configEpoch) {
    clusterHandleConfigEpochCollision(sender);
}
```

```c
void clusterHandleConfigEpochCollision(clusterNode *sender) {
    /* Prerequisites: nodes have the same configEpoch and are both primaries. */
    if (sender->configEpoch != myself->configEpoch ||
        !clusterNodeIsPrimary(sender) || !clusterNodeIsPrimary(myself))
        return;
    /* Don't act if the colliding node has a smaller Node ID. */
    if (memcmp(sender->name, myself->name, CLUSTER_NAMELEN) <= 0) return;
    /* Get the next ID available at the best of this node knowledge. */
    server.cluster->currentEpoch++;
    myself->configEpoch = server.cluster->currentEpoch;
    clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG | CLUSTER_TODO_FSYNC_CONFIG
                        | CLUSTER_TODO_BROADCAST_ALL);
    serverLog(LL_NOTICE,
        "configEpoch collision with node %.40s (%s). configEpoch set to %llu",
        sender->name, humanNodename(sender),
        (unsigned long long)myself->configEpoch);
}
```

The node with the **larger node ID** increments its epoch and rebroadcasts. The smaller-ID node stays at `N`. Eventually the larger-ID node is at `N+1` and `clusterUpdateSlotsConfigWith()` can resolve ownership.

This WILL eventually converge, which is why the comment in `clusterBumpConfigEpochWithoutConsensus()` says "the cluster will not explode." However:

- It fires only once per PING/PONG cycle, not proactively
- The window between partition heal and collision resolution is the observable split-brain
- During this window, writes to the conflicting slots are split between two nodes

The code's own comment (`cluster_legacy.c:2437`) acknowledges this risk: "nothing is worse than a split-brain condition in a distributed system."

```
cluster_legacy.c:8159  TAKEOVER branch
```

The condition that was not met: `myself->configEpoch != 0 AND myself->configEpoch == maxEpoch`.

The precondition that set this up: R1's `configEpoch` was left at a non-zero value (equal to P1's `configEpoch`) by a prior failover cycle that was later reversed at runtime via `clusterSetPrimary()`, which does not reset `myself->configEpoch`.

The propagation failure: `clusterUpdateSlotsConfigWith()` uses a strict `<` comparison on `configEpoch`, so equal epochs prevent slot transfer in either direction until `clusterHandleConfigEpochCollision()` runs and pushes one node ahead.
