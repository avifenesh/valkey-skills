## Root Cause Analysis: Split-Brain After Partition Healing

### The Only Place `currentEpoch` Increments During Automatic Failover

`cluster_legacy.c:5767`, inside `clusterHandleReplicaFailover()`:

```c
/* Ask for votes if needed. */
if (server.cluster->failover_auth_sent == 0) {
    server.cluster->currentEpoch++;
    server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
    clusterRequestFailoverAuth();
    server.cluster->failover_auth_sent = 1;
    clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG | CLUSTER_TODO_UPDATE_STATE | CLUSTER_TODO_FSYNC_CONFIG);
    return; /* Wait for replies. */
}
```

This is the single increment point. There is no other path that increments `currentEpoch` during an automatic election. The TAKEOVER path (`clusterBumpConfigEpochWithoutConsensus`, line 2389) is a separate code path requiring explicit operator invocation.

---

### Conditions That Must ALL Be Met to Reach Line 5767

`clusterHandleReplicaFailover()` is called from both `clusterCron()` (the time event, ~10/sec) and `clusterBeforeSleep()`. The function returns early at seven checkpoints before reaching the increment:

**Guard 1 - Basic role check (`cluster_legacy.c:5616-5623`):**
```c
if (clusterNodeIsPrimary(myself) || myself->replicaof == NULL ||
    (!nodeFailed(myself->replicaof) && !manual_failover) ||
    (server.cluster_replica_no_failover && !manual_failover)) {
    server.cluster->cant_failover_reason = CLUSTER_CANT_FAILOVER_NONE;
    return;
}
```
Three conditions embedded here:
- We must be a replica
- We must have a known primary (`replicaof != NULL`)
- The primary must be in hard FAIL state (not just PFAIL) OR this is a manual failover

The `nodeFailed()` macro checks `CLUSTER_NODE_FAIL` specifically. **PFAIL alone is not enough.** This is the most critical gate.

**Guard 2 - Data age check (`cluster_legacy.c:5642-5648`):**
```c
if (server.cluster_replica_validity_factor &&
    data_age > (((mstime_t)server.repl_ping_replica_period * 1000) +
                (server.cluster_node_timeout * server.cluster_replica_validity_factor))) {
    if (!manual_failover) {
        clusterLogCantFailover(CLUSTER_CANT_FAILOVER_DATA_AGE);
        return;
    }
}
```
The replica's data must be recent enough. A long partition means a large `data_age`, and if `cluster-replica-validity-factor` is nonzero (default: 10), the replica silently refuses to start an election.

**Guard 3 - Election timer setup (`cluster_legacy.c:5653-5711`):**
The first time through after the retry window expires, `failover_auth_time` is set to `now + delay + jitter + rank_offset`. The function then returns immediately:
```c
if (now < server.cluster->failover_auth_time) return;
```

**Guard 4 - Rank re-check (`cluster_legacy.c:5725-5757`):**
If the replica's rank changed (a sibling has better replication offset), the delay is extended. Another early return:
```c
if (now < server.cluster->failover_auth_time) {
    clusterLogCantFailover(CLUSTER_CANT_FAILOVER_WAITING_DELAY);
    return;
}
```

**Guard 5 - Election timeout (`cluster_legacy.c:5759-5763`):**
```c
if (auth_age > auth_timeout) {
    clusterLogCantFailover(CLUSTER_CANT_FAILOVER_EXPIRED);
    return;
}
```
`auth_timeout = max(cluster_node_timeout * 2, 2000)`. If the election window has expired without quorum, this fires and the whole attempt is abandoned until `auth_retry_time = auth_timeout * 2` passes.

**Guard 6 - Already sent (`cluster_legacy.c:5766`):**
```c
if (server.cluster->failover_auth_sent == 0) {
```
Once a vote request has been sent, the function falls through to the quorum check instead of sending another. The epoch is only incremented ONCE per election attempt.

### How `nodeFailed()` Gets Set - The FAIL Propagation Chain

For Guard 1 to pass, P1 must be in hard FAIL state on R1. This requires:

1. `clusterCron()` marks P1 as PFAIL when `node_delay > cluster_node_timeout` (`cluster_legacy.c:5616-5623` region, earlier in cron)
2. Gossip carries the PFAIL flag to other nodes, which add failure reports via `clusterProcessGossipSection()`
3. `markNodeAsFailingIfNeeded()` (`cluster_legacy.c:2585-2604`) counts reports and checks quorum:

```c
void markNodeAsFailingIfNeeded(clusterNode *node) {
    int failures;
    int needed_quorum = (server.cluster->size / 2) + 1;

    if (!nodeTimedOut(node)) return;  // Must be PFAIL
    if (nodeFailed(node)) return;     // Already FAIL

    failures = clusterNodeFailureReportsCount(node);
    if (clusterNodeIsVotingPrimary(myself)) failures++;
    if (failures < needed_quorum) return;

    markNodeAsFailing(node);
    clusterSendFail(node->name);  // Broadcast FAIL to the entire cluster
}
```

The quorum is `(server.cluster->size / 2) + 1` where `size` counts ALL primaries with at least one slot across ALL known nodes - including those unreachable. In a 3-primary cluster: quorum = 2.

`clusterSendFail()` broadcasts a `CLUSTERMSG_TYPE_FAIL` message to every connected node. Any node receiving this immediately sets `CLUSTER_NODE_FAIL` for that node.

### What Prevented the Epoch From Advancing (Guard 1 Failure)

The failover during the partition would have incremented the epoch normally - **the epoch DID advance**. R1 incremented from N to N+1. The reason both conflicting nodes now show the same `currentEpoch` is convergence, not stasis. When P1 reconnected, `clusterProcessPacket` at line 3902-3903 ran on every message P1 received:

```c
sender_claimed_current_epoch = ntohu64(msg->currentEpoch);
if (sender_claimed_current_epoch > server.cluster->currentEpoch)
    server.cluster->currentEpoch = sender_claimed_current_epoch;
```

P1's `currentEpoch` was updated to N+1 from the first PING it received after reconnection. Both nodes showing the same `currentEpoch` is the expected post-partition state. The diagnostic observation of "same epoch" is not the anomaly - the overlapping slot ownership is.

### Why P1 Is Still Claiming Its Slots After Partition Heals

This is the actual bug. Two mechanisms should have demoted P1. Here is why each can fail.

**Mechanism 1: The UPDATE message path**

When P2 or P3 receives P1's post-reconnection PING claiming the contested slots, `clusterProcessPacket` runs the reverse check at `cluster_legacy.c:4317-4358`:

```c
bool found_new_owner = false;
for (size_t w = 0; w < CLUSTER_SLOT_WORDS && !found_new_owner; w++) {
    ...
    clusterNode *slot_owner = server.cluster->slots[slot];
    if (slot_owner == sender || isSlotUnclaimed(slot)) continue;
    if (slot_owner->configEpoch > sender_claimed_config_epoch) {
        clusterSendUpdate(sender->link, slot_owner);
        found_new_owner = true;
        break;
    }
}
```

Note the `break` on `found_new_owner = true`. Only ONE UPDATE is sent per PING, for only ONE slot. If P1 owns hundreds of slots, the sender learns about one new owner per gossip round. This is slow but eventually converges.

But the critical failure: **`clusterSendUpdate` at line 5056 silently drops if the link is null:**
```c
void clusterSendUpdate(clusterLink *link, clusterNode *node) {
    if (link == NULL) return;
```

During the exact window when P1 reconnects and sends its first PING, P1 is `sender` with an established inbound link from P1's side, but the outbound link from P2 TO P1 may not yet be established. `clusterCron()` handles outbound link creation asynchronously. In that window: inbound PING from P1 arrives, UPDATE is computed, but `sender->link` (the outbound link from P2 to P1) is still NULL. The UPDATE is dropped silently.

On the next gossip cycle, the link is established and the UPDATE goes through. This explains a brief transient split-brain but not a persistent one.

**Mechanism 2: configEpoch collision resolution - THE PERSISTENT FAILURE PATH**

Here is the exact chain that creates the persistent state you're observing.

P1 reconnects and sends a PING. Before or concurrently with the UPDATE being processed, P2 or P3 may receive P1's PING and run `clusterHandleConfigEpochCollision` (line 4364-4366):

```c
if (sender && nodeIsPrimary(myself) && nodeIsPrimary(sender) &&
    sender_claimed_config_epoch == myself->configEpoch) {
    clusterHandleConfigEpochCollision(sender);
}
```

But this fires based on P1 and the RECEIVING node (P2 or P3) having equal configEpochs, not P1 and R1 colliding. This is not the path.

The collision that matters happens when P1's PING reaches R1 directly, and R1 has `configEpoch = N+1` for those slots. R1 is now a primary (was promoted). R1 receives P1's PING claiming the same slots with configEpoch N+1 (if P1 had its configEpoch bumped to N+1 during the partition independently, e.g., through a resharding close operation or TAKEOVER of an unrelated shard).

When P1 and R1 both have `configEpoch = N+1`:

```c
// cluster_legacy.c:2449-2461
void clusterHandleConfigEpochCollision(clusterNode *sender) {
    if (sender->configEpoch != myself->configEpoch || !clusterNodeIsPrimary(sender) || 
        !clusterNodeIsPrimary(myself)) return;
    if (memcmp(sender->name, myself->name, CLUSTER_NAMELEN) <= 0) return;
    server.cluster->currentEpoch++;
    myself->configEpoch = server.cluster->currentEpoch;
    clusterDoBeforeSleep(CLUSTER_TODO_SAVE_CONFIG | CLUSTER_TODO_FSYNC_CONFIG | CLUSTER_TODO_BROADCAST_ALL);
}
```

The node with the lexicographically larger node ID increments its own configEpoch. Say P1's node ID > R1's node ID: P1 gets `configEpoch = N+2`. Now P1's configEpoch is strictly greater than R1's. When `clusterUpdateSlotsConfigWith` subsequently tries to transfer P1's slots to R1, the check at line 3079 fails:

```c
if (isSlotUnclaimed(j) ||
    server.cluster->slots[j]->configEpoch < senderConfigEpoch ||  // P1 has N+2, R1 sends N+1
    clusterSlotFailoverGranted(j)) {
```

`N+2 < N+1` is false. **The slot is not transferred.** P1 keeps its slots permanently. P1 is now the de-facto owner in the cluster topology map for those slots, and R1 is also claiming them from its own perspective. `CLUSTER NODES` shows both as slot owners. `cluster_state:ok` because all slots ARE covered (just doubly so).

---

### The Specific Condition Not Met

The condition not met is at `cluster_legacy.c:3079`:

```c
server.cluster->slots[j]->configEpoch < senderConfigEpoch
```

This condition must be strictly true for slot ownership to transfer. It evaluates as false when P1's configEpoch (elevated to N+2 by `clusterHandleConfigEpochCollision`) exceeds R1's configEpoch (N+1 from the election).

The collision resolution at line 2454 is the proximate cause:
```c
if (memcmp(sender->name, myself->name, CLUSTER_NAMELEN) <= 0) return;
```

This check was designed to make only the higher-ID node bump its epoch (so there is a unique winner). But the design assumes the higher-ID node is R1 (the legitimate winner). If instead P1 has the higher node ID, P1 bumps itself to N+2 and becomes the winner - **inverting the intended outcome of the election**.

### Remediation

**Immediate**: On P1, manually run `CLUSTER FAILOVER TAKEOVER` from R1. This calls `clusterBumpConfigEpochWithoutConsensus` which sets R1's configEpoch above P1's current N+2, then `clusterFailoverReplaceYourPrimary` forces P1 to accept R1 via the next UPDATE cycle.

**Root diagnosis**: Check P1's `cluster.log` for any `clusterBumpConfigEpochWithoutConsensus` or `configEpoch set to` log entries during the partition window. That event is what put P1's configEpoch at the same value as R1's election epoch, which then allowed the collision resolution to invert the failover result.
