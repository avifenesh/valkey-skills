Analysis written to `ANALYSIS.md`. Here's the summary:

**Root Cause**: The `Dockerfile.buggy-valkey` patches out `server.cluster->currentEpoch++` on line ~5484 of `src/cluster_legacy.c` in Valkey 9.0.3. This line is inside `clusterHandleReplicaFailover()`, in the block guarded by `failover_auth_sent == 0`, where a replica increments the cluster epoch before requesting failover votes.

**Why it causes split-brain**: Without the increment, the new master gets `configEpoch = currentEpoch` (the old value) instead of `currentEpoch + 1`. If the old master's `configEpoch` equals `currentEpoch`, both masters end up with the **same** epoch. The slot-rebinding check (`configEpoch < senderConfigEpoch`) fails because N is not < N, so the old master never steps down. Two nodes claim the same slots.

**In the test run**: The bug was masked because valkey-1 had `configEpoch = 1` while `currentEpoch` was already `6` (from initial 6-node setup). So 6 > 1 was enough for the old master to step down. The split-brain manifests when the failing master has `configEpoch == currentEpoch`.

**Fix**: Restore the `server.cluster->currentEpoch++` line, or simply remove the `sed` patch from the Dockerfile to use stock Valkey 9.0.3.

I also fixed two issues in `docker-compose.yml` that prevented the reproduce script from running (duplicate `valkey-server` in the command conflicting with the ENTRYPOINT, and missing `--protected-mode no`).
