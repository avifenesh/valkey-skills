The bug has been found, fixed, and verified. Here's the summary:

**The Bug:** A bogus function `clusterShouldDeferEpochBump()` was added to `src/cluster_legacy.c:2457-2477` that permanently blocks epoch collision resolution after any failover. It checks if any primary's `configEpoch == currentEpoch`, which is **always true** after failover (since that's how failover works). This prevents the cluster from resolving which node owns the contested slots, causing a permanent split-brain.

**The Fix:** Removed the `clusterShouldDeferEpochBump()` function and its call in `clusterHandleConfigEpochCollision()`, restoring the standard epoch collision resolution algorithm that uses node ID comparison to deterministically pick a winner.

**Verification:** After the fix, `reproduce.sh` shows the old primary correctly stepping down to slave, with only one master per slot range, `cluster_state: ok`, and all 16384 slots covered.
