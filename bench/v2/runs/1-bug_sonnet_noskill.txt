`ANALYSIS.md` has been written. Here's a summary of the findings:

---

## Root Cause

The Dockerfile injects a bug by commenting out **one line** in `src/cluster_legacy.c` (~line 5483), inside `clusterHandleReplicaFailover()`:

```c
/* server.cluster->currentEpoch++; */   // ← removed by the patch
server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
```

## Why This Causes Split-Brain

Valkey cluster resolves slot-ownership conflicts via `configEpoch`: the node with the higher epoch wins. The design invariant is:

> **A successful failover must produce a new primary whose `configEpoch` is strictly greater than every other `configEpoch` in the cluster.**

The `currentEpoch++` increment is the mechanism that enforces this. Without it:

1. **Epoch collision.** With 3 primaries having configEpochs 1/2/3 and `currentEpoch = 3`, the promoted replica gets `configEpoch = 3` — the **same** as Primary 3. Two primaries now share an epoch.

2. **Non-deterministic collision resolution.** `clusterHandleConfigEpochCollision` fixes this asynchronously, but only the node with the lexicographically higher node ID increments its epoch. The resolution takes gossip rounds.

3. **Old primary rejoins during the ambiguity window.** The formerly partitioned primary reconnects while the collision is still in flight. Because `currentEpoch` was never bumped, the whole cluster still shows `currentEpoch = 3`. There is no unambiguous "higher authority" signal; the old primary does not demote cleanly.

4. **`lastVoteEpoch` blocks retries.** Voters set `lastVoteEpoch = 3`. Any retry election at the same epoch is blocked, preventing self-healing via re-election.

## Fix

Restore the removed line:
```c
server.cluster->currentEpoch++;
```

## Workaround (live cluster)

On the correct new primary: `CLUSTER FAILOVER FORCE` or `CLUSTER BUMPEPOCH`  
On the old partitioned primary that should step down: `CLUSTER RESET SOFT` then `CLUSTER REPLICATE <new-primary-id>`
