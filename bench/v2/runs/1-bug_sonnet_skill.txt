Here's what I found:

## Root Cause

The `Dockerfile.buggy-valkey` patches a single line out of `src/cluster_legacy.c` in the `clusterHandleReplicaFailover()` function:

```c
/* This line is commented out in the buggy build: */
/* server.cluster->currentEpoch++; */
server.cluster->failover_auth_epoch = server.cluster->currentEpoch;
clusterRequestFailoverAuth();
```

## Why This Breaks Failover

Valkey's split-brain prevention relies entirely on **epoch ordering**:

1. When a replica initiates a failover election, it must first increment `currentEpoch` (the cluster-wide logical clock).
2. The `failover_auth_epoch` is set to this new, higher value.
3. When the election is won, the new primary sets its `configEpoch = failover_auth_epoch` — a value **strictly greater** than the old primary's `configEpoch`.
4. When the old primary reconnects, it sees the new primary's higher `configEpoch` for the same slots and steps down automatically.

With the increment removed, `failover_auth_epoch` stays at the **same** value as the current epoch. The promoted replica's `configEpoch` ends up equal to (or not reliably greater than) the old primary's `configEpoch`. When the partition heals, neither node sees a higher authority and both continue claiming ownership of the same slots — permanent split-brain.

## Fix

Remove the `sed` patch from the Dockerfile so the upstream `server.cluster->currentEpoch++;` line is compiled in. That single line is the entire safeguard.

Full analysis with the exact code locations, epoch arithmetic, `CLUSTER NODES` output interpretation, and a manual workaround is in `ANALYSIS.md`.
