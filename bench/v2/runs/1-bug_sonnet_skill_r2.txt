## What was found and fixed

### Bug 1 (C source — the root cause): `src/cluster_legacy.c`

A fabricated function `clusterShouldDeferEpochBump()` was inserted before `clusterHandleConfigEpochCollision()`. It returned 1 (defer) whenever any primary's `configEpoch == currentEpoch`. After a successful failover, the promoted primary **always** has `configEpoch == currentEpoch` — that's how failover works. So this guard permanently blocked epoch collision resolution, leaving the cluster in split-brain indefinitely.

**Fix:** Removed `clusterShouldDeferEpochBump()` and its call entirely. The original collision resolution logic (node with larger Node ID bumps its epoch) is correct and sufficient.

### Bug 2 (Dockerfile): `ENTRYPOINT ["valkey-server"]`

The Dockerfile's `ENTRYPOINT` + docker-compose `command: valkey-server --port ...` caused double invocation: `valkey-server valkey-server --port ...`, crashing immediately.

**Fix:** Removed the `ENTRYPOINT` line.

### Bug 3 (Dockerfile): Missing `libvalkeylua.so` in runtime image

The Makefile compiled the server with `-DLUA_ENABLED` by default, requiring `libvalkeylua.so` at runtime. The Dockerfile didn't copy that `.so` to the final image, causing a panic on startup.

**Fix:** Added `BUILD_LUA=no` to the make command.
