**TaskQueue** (lines 23-40): Produces tasks via `xadd`, with `addBatch` for bulk enqueue.

**Worker** (lines 45-210): 
- `ensureGroup()` - creates the consumer group idempotently (catches `BUSYGROUP`)
- `start()` - polling loop using `XREADGROUP` via `customCommand`, processes messages, ACKs them, auto-stops after ~5s idle
- `processMessage()` - simulates 50-200ms random work
- `claimStale()` - uses `XPENDING ... IDLE 30000` to find stale messages; moves to DLQ after 3 retries, otherwise claims and reprocesses
- `moveToDLQ()` - reads original message via `XRANGE`, adds to dead-letter stream with metadata, ACKs original
- `_parseFields()` - converts flat `[k, v, k, v]` arrays to objects

**Dashboard** (lines 215-261): Shows stream length, pending count per consumer, DLQ count, and total processed every 5 seconds.

**Main** (lines 266-337):
- Connects `GlideClient` to Valkey
- Enqueues 100 tasks with high/medium/low priorities
- Starts 3 concurrent workers
- Dashboard on 5s interval
- SIGTERM/SIGINT graceful shutdown (stops workers, waits for completion, prints final status)
- Workers auto-stop after processing all messages (25 consecutive empty reads)

All stream operations use `@valkey/valkey-glide` — native methods for `xadd`/`xlen`/`xack`/`xgroupCreate` and `customCommand` for `XREADGROUP`/`XPENDING`/`XCLAIM`/`XRANGE`/`XADD` (DLQ) to handle complex response parsing reliably.
