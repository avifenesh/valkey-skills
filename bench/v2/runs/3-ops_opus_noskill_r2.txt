All files are in place. Here's a summary of what was created:

### Deliverables

| File | Purpose |
|---|---|
| `kind-config.yaml` | Kind cluster with 1 control-plane + 3 workers (for anti-affinity spread) |
| `helm-values.yaml` | Bitnami `valkey-cluster` Helm chart values with all requirements configured |
| `valkey-cluster-cr.yaml` | Custom Resource definition for the valkey-operator (reference/migration target) |
| `deploy.sh` | End-to-end deployment: kind cluster, TLS certs, Helm install, ACL lockdown |
| `test.sh` | Comprehensive validation: connectivity, write/read, vector search, ACL restrictions |

### Key design decisions

**Cluster topology**: 6 pods (3 primaries + 3 replicas), soft anti-affinity across 3 kind worker nodes.

**Security/ACL**: Three users configured in `commonConfiguration`:
- `admin` — `+@all` (full access)
- `app` — `+@all -@admin -@dangerous` (data operations only)
- `monitor` — `+info +cluster|info +client|list +ping +command` (read-only monitoring)
- Default user is disabled post-init via `ACL SETUSER default off`

**TLS**: Self-signed CA + server cert with proper SANs for all cluster service names. Custom probes authenticate as `admin` so they survive default-user disablement.

**HA**: PDB with `maxUnavailable: 1` guarantees at least 2 of 3 primaries are always available.

**Storage**: PVCs with AOF (`appendonly yes`, `appendfsync everysec`).

**Monitoring**: Prometheus exporter sidecar on port 9121 with `prometheus.io/scrape` annotations.

**Vector search**: Module loading is attempted post-deploy; if the image includes `valkeysearch.so`, the test runs a full FT.CREATE + KNN search via a Python helper pod.
