All files are in place. Here's a summary of what was created and the key design decisions:

## Deliverables

| File | Purpose |
|---|---|
| `kind-config.yaml` | 1 control-plane + 3 worker nodes (3 workers → each primary on its own node) |
| `docker/Dockerfile` | Two-stage build: extracts `valkeysearch.so` from `valkey/valkey-bundle` into `bitnami/valkey` |
| `helm/values.yaml` | Bitnami `valkey-cluster` chart values |
| `manifests/metrics-service.yaml` | Headless Service for Prometheus scraping |
| `deploy.sh` | End-to-end deploy script |
| `test.sh` | 13 validation tests |

## Key decisions

**Chart choice: Bitnami `valkey-cluster`** (not an operator)
- Hyperspike operator has issue [#186](https://github.com/hyperspike/valkey-operator/issues/186): creates extra primaries instead of per-shard replicas — unsafe for a 3+3 topology
- SAP operator: Sentinel only, no cluster mode
- Official Valkey chart: no cluster mode
- Bitnami `valkey-cluster`: the only correct fit for 3-primary + 3-replica cluster mode, with PDB/TLS/metrics/anti-affinity built in

**valkey-search via `valkey/valkey-bundle`**
The `valkey/valkey-bundle` image ships all official modules (search 1.2.0, json, bloom) at `/opt/valkey/modules/`. The Dockerfile copies `valkeysearch.so` from the bundle into `bitnami/valkey` so the Bitnami chart's startup scripts still work.

**ACL design**
Default user is kept enabled (required for Bitnami startup scripts and inter-node replication via `masterauth`) but stripped of key-space access — only cluster/replication commands allowed. Application code must authenticate as `admin`, `app`, or `monitor`.

**TLS**: self-signed CA + server cert (SAN wildcard covers all pod DNS names), generated by `deploy.sh` and stored in `Secret valkey-tls`.
