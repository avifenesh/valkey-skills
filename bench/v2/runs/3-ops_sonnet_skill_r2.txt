### Design choices

**Chart**: Bitnami `valkey-cluster` (`oci://registry-1.docker.io/bitnamicharts/valkey-cluster`) — chosen because the official Valkey chart has no cluster-mode support, and the official operator is explicitly not production-ready yet.

**Cluster**: `cluster.nodes:6, cluster.replicas:1` = 3 primaries + 3 replicas.

**valkey-search module**: compiled in a multi-stage Dockerfile and injected via an init container into a shared `emptyDir` volume. `loadmodule /modules/valkey-search.so` is appended via `extraConfig`.

**TLS**: self-signed CA + server cert with SAN covering all pod DNS names; generated by `deploy.sh` with `openssl` and stored as `valkey-tls-secret`.

**ACL**: 4 users in `valkey-acl-secret` (mounted as `aclfile`). The `default` user is restricted to `nokeys` + cluster-internal commands only (can't access application data). A `replication` user handles inter-node auth via `masteruser replication` in `extraConfig` — this is a Valkey 7+ feature (`masteruser`/`primaryuser` config option).

**PDB**: `maxUnavailable:1` across all 6 pods guarantees at least 2 primaries survive any single disruption.

**Anti-affinity**: `podAntiAffinityPreset:soft` — spreads pods across the 3 workers without blocking scheduling.

**Metrics**: `redis_exporter` sidecar enabled; Service annotated for `prometheus.io/scrape`.
