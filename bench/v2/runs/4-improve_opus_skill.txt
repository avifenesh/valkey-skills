Done. I've written assessments for all 12 scenarios to `ANSWERS.md`. Here's a summary of the findings:

| # | Scenario | Verdict | Core Issue |
|---|----------|---------|------------|
| 1 | Session Management | Problematic | `KEYS` blocks the server; use `SET ... EX` + `EXPIRE` to extend TTLs |
| 2 | Product Catalog Cache | Problematic | `DEL` blocks main thread; use `UNLINK` (async) with batching |
| 3 | Field-Level TTL | Problematic | Valkey 7.4+ has `HEXPIRE`; no need for separate keys |
| 4 | Rate Limiting | Problematic | Non-atomic ops + wrong ordering; use a Lua script |
| 5 | Distributed Lock | Problematic | GET-then-DEL race condition; use atomic Lua check-and-delete |
| 6 | Real-time Inventory | Problematic | `KEYS` on every page load; use a hash per SKU instead |
| 7 | Job Queue | Problematic | `BRPOPLPUSH` deprecated, no ack tracking; use Streams + consumer groups |
| 8 | Conditional Updates | Problematic | WATCH retries under contention; use Lua compare-and-set |
| 9 | Full-text Search | Problematic | Full keyspace scan; build inverted index with sets or use Search module |
| 10 | Pub/Sub Notifications | Problematic | Fire-and-forget loses messages; use Streams with consumer groups |
| 11 | Monitoring | Problematic | Threshold too high + missing client names; lower threshold, use `CLIENT SETNAME`, add latency monitoring |
| 12 | Leaderboard TTL | Problematic | `EXPIRE` works on any key type including sorted sets; just set it |
