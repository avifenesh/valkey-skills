Here's a summary of all 9 fixes:

| # | Anti-Pattern | Fix | Why |
|---|---|---|---|
| **1** | No timeout or reconnect config | Added `requestTimeout(5000)`, `BackoffStrategy` with exponential backoff, and a bounded `get(10, SECONDS)` on connection | Production connections need timeouts to fail fast and auto-reconnect on transient failures |
| **2** | `DEL` (blocking delete) | Changed to `UNLINK` | UNLINK removes the key reference in O(1) on the main thread and frees memory in a background thread, avoiding latency spikes on large keys |
| **3** | N individual `GET` calls | Single `MGET` call | Reduces N network round-trips to 1. GLIDE auto-pipelines, but MGET is still semantically cleaner and guaranteed atomic |
| **4** | N sequential `SET` calls | `Batch(false)` pipeline | Sends all SETs in one batch. Even with GLIDE's auto-pipelining, explicit batching guarantees all commands travel together |
| **5** | `KEYS product:*` | `SCAN` with cursor iteration | KEYS blocks the server while scanning the entire keyspace. SCAN iterates incrementally, allowing other commands between batches |
| **6** | `SET` without TTL on cache | `SetOptions.expiry(Seconds(3600L))` | Cache entries without TTL grow unbounded until OOM. 1-hour TTL ensures stale data is reclaimed |
| **7** | `SET` without TTL on sessions | `SetOptions.expiry(Seconds(1800L))` | Sessions without TTL leak memory forever. 30-minute TTL matches standard session idle timeout |
| **8** | `SORT` on a list | `ZRANGE leaderboard 0 99 REV` on a sorted set | SORT is O(N + M*log M) and blocks the server. Sorted sets maintain order on insert — ZRANGE is O(log N + M) |
| **9** | KEYS + individual GETs for counting | SCAN + batched MGET per scan page | Eliminates server-blocking KEYS and replaces N GETs with batched MGET per scan page |

Additional improvements:
- Class implements `AutoCloseable`, `main` uses try-with-resources
- `client` field is `final`
- Constants for TTL values and scan batch size
- Early returns on empty inputs
