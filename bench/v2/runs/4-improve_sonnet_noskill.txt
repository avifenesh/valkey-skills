### Changes made

**1. Wrong client class** (`GlideClusterClient` → `GlideClient`)
The docker-compose runs a single standalone node. `GlideClusterClient` is for Valkey Cluster mode only.

**2. `KEYS` → cursor-based `SCAN`** (4 occurrences replaced by `scanKeys()`)
`KEYS product:*` is a blocking O(N) server-wide scan that stalls every other client. `SCAN` iterates in batches without blocking.

**3. N+1 individual GETs → `mget`**
`/products`, `/products/search`, `/products/category/:cat`, and `/products/expired` all fetched one key at a time. Now each does a single `mget(keys)` round-trip.

**4. Sequential `for await set()` → `mset`**
`/products/bulk` and `/cache/warm` looped with `await client.set()` per item. Now a single `mset(keyValueMap)` call handles the whole batch.

**5. Batched `del` for expired cleanup**
Previously deleted one key per loop iteration. Now collects all expired keys and issues one `client.del(expiredKeys)`.

**6. Route ordering bug fixed**
`/products/search`, `/products/category/:cat`, and `/products/expired` were registered *after* `/:id`. Express matched them as `id = "search"` etc. Specific routes now come first.

**7. Error handling** — all handlers wrapped in `try/catch`, returning `500` instead of crashing.

**8. Input validation** — `GET /products/search` returns `400` when `q` is missing.

**9. Env-var config** — `VALKEY_HOST`/`VALKEY_PORT` env vars with `localhost:6379` defaults (required for containerized deployments).

**10. Graceful shutdown** — `SIGTERM` handler closes the client connection cleanly.
