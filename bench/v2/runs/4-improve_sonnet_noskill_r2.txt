I've completed the assessment of all 12 Valkey usage scenarios and written the answers to ANSWERS.md.

## Key findings:

**Critical issues requiring immediate attention:**
1. **Session cleanup (#1)** - `KEYS session:*` blocks production, use native TTL instead
2. **Inventory lookup (#6)** - `KEYS` on every page load is catastrophic, switch to Hash structure  
3. **Product search (#9)** - `KEYS product:*` with app-layer filtering is unscalable, need RediSearch or external search

**High-priority fixes:**
- **Lock release (#5)** - Race condition between GET and DEL, use Lua script for atomicity
- **Pub/Sub (#10)** - Message loss during disconnects, migrate to Streams for persistence
- **Leaderboards (#12)** - Incorrect assumption that sorted sets can't have TTL (they can!)

**Performance optimizations:**
- **Cache invalidation (#2)** - Batch DEL operations or use UNLINK
- **Rate limiting (#4)** - Counter-based approach more memory-efficient than sorted sets
- **Job queue (#7)** - Use BLMOVE, fix cleanup logic, or migrate to Streams
- **Conditional updates (#8)** - Lua scripts eliminate retry loops under contention

Each answer includes the specific Valkey commands and data structures needed for implementation.
