Here's a summary of every change and why:

---

### Fix 1 — Constructor: resilient connection config
- Added `requestTimeout(5000)` — prevents commands from hanging indefinitely on a dead connection.
- Added `reconnectStrategy(BackoffStrategy)` — exponential backoff with jitter (100ms base, 2x factor, 5 retries, 20% jitter) so the client recovers automatically from transient failures instead of failing hard.
- Made `client` field `final` — signals immutability and prevents accidental reassignment.
- Added `implements AutoCloseable` — enables try-with-resources in callers.

### Fix 2 — `deleteProduct`: `DEL` → `UNLINK`
`DEL` frees memory synchronously on the main thread. For large keys this blocks all other clients. `UNLINK` removes the key reference in O(1) and defers memory reclamation to a background thread. It's strictly better; the only reason to use `DEL` is if you need synchronous deletion guarantees (rare).

### Fix 3 — `getProducts`: N individual GETs → `MGET`
The loop issued one network round-trip per product. `MGET` fetches all N values in a single command. For 20 products this is a 20× reduction in round-trips.

### Fix 4 — `saveProducts`: N individual SETs → `MSET`
Same problem as Fix 3 on the write side. `MSET` writes all pairs atomically in one command.

### Fix 5 — `getAllProductKeys`: `KEYS` → `SCAN`
`KEYS pattern` blocks the Valkey server thread while it scans the entire keyspace — on a production instance with millions of keys this freezes all clients for seconds. `SCAN` iterates in small incremental batches (cursor-based). It may return duplicates, which a `LinkedHashSet` deduplicates while preserving insertion order.

### Fix 6 — `cacheProduct`: SET without TTL → `SET EX 3600`
Cache entries without expiry live forever. Memory fills with stale data until the server hits `maxmemory` and starts evicting (or crashes). Every cache write must carry a TTL.

### Fix 7 — `saveSession`: SET without TTL → `SET EX 1800`
Same as Fix 6. Sessions that never expire are a memory leak and a security risk — stale session tokens remain valid indefinitely.

### Fix 8 — `getLeaderboard`: blocking `SORT` → `ZREVRANGE` on sorted set
`SORT` is O(N log N) and holds the server thread for the full duration — dangerous on any non-trivial dataset. The correct data structure for a leaderboard is a **sorted set**: `ZADD` maintains members in score order, and `ZREVRANGE` is O(log N + M). Added `addToLeaderboard()` to show how to populate the sorted set.

### Fix 9 — `countProductsByCategory`: KEYS + per-key GET → SCAN + batched MGET
The original did two anti-patterns at once: `KEYS` (server-blocking) followed by N individual `GET` calls (N round-trips). The fix uses SCAN (Fix 5) and batches up to 1000 keys per `MGET`. The code comment also notes the real production answer: maintain a per-category counter key (`INCR counter:category:<name>`) at write time so counting is O(1) at read time.

### Fix 10 — `main`: try-with-resources
Wraps `ProductCache` in try-with-resources so `close()` is guaranteed to run even if an exception is thrown mid-demo, preventing connection leaks.
