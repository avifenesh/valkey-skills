# Valkey GLIDE - The Official Multi-Language Client for Valkey

A practical guide for developers using the GLIDE (General Language Independent Driver for the Enterprise) client library to interact with Valkey and Redis OSS.

---

## 1. What is Valkey GLIDE?

Valkey GLIDE is the official open-source client library for the Valkey datastore. It provides a unified, high-performance interface across multiple programming languages - Python, Java, Node.js, Go, PHP, and C# - all powered by a shared Rust core engine.

GLIDE supports:
- **Valkey** versions 7.2, 8.0, 8.1, 9.0
- **Redis OSS** versions 6.2, 7.0, 7.1, 7.2

The name stands for **G**eneral **L**anguage **I**ndependent **D**river for the **E**nterprise.

### Why GLIDE Exists

Before GLIDE, each language had its own independent client (redis-py, ioredis, Jedis, go-redis, etc.) with varying levels of reliability, cluster handling, and feature completeness. GLIDE provides a single, consistently-behaved client across all languages, with enterprise-grade reliability features built into a shared Rust core.

### Key Capabilities

- **Cluster-aware routing** - automatic topology discovery, slot-based routing, multi-slot command splitting
- **AZ Affinity** - availability-zone-aware read routing for reduced latency and cross-AZ cost savings (Valkey 8.0+)
- **Automatic reconnection** - exponential backoff with jitter, PubSub resubscription on reconnect
- **Batching** - both atomic (transactions) and non-atomic (pipelines) in a unified API
- **OpenTelemetry** - integrated tracing and metrics with per-command spans
- **Multiplexed connections** - single connection per node, pipelining all requests
- **Lua scripting** - automatic script caching via SHA1 / EVALSHA

## 2. Architecture

### Three-Layer Design

```
Application Code (Python / Java / Node.js / Go / PHP / C#)
        |
Language Wrapper (lightweight bindings)
        |
    FFI / CGO / napi-rs / JNI / Protobuf
        |
    Rust Core Engine (glide-core)
        |
    Valkey / Redis OSS Server
```

**Rust Core (glide-core)** handles all heavy lifting:
- RESP protocol parsing and message encapsulation
- Connection establishment, multiplexing, and reconnection
- Cluster topology discovery and tracking
- Slot-based command routing
- Error management and retry logic
- Inflight request limiting (default: 1000 per connection)
- OpenTelemetry span and metric emission

**Language Wrappers** are thin, idiomatic interfaces:
- Python: async (asyncio/anyio/trio) and sync APIs
- Java: CompletableFuture-based async API
- Node.js: Promise-based async with TypeScript/CommonJS/ESM support
- Go: synchronous API with goroutine safety via CGO bridge
- PHP: synchronous via FFI extension
- C#: async/await with .NET 8.0+

**Communication Layer** between wrappers and core varies by language:
- Node.js uses napi-rs (native Rust bindings for Node)
- Java uses JNI with Protobuf serialization for all commands
- Python uses Protobuf (with URI-based creation to reduce Protobuf dependency)
- Go uses CGO with C headers generated by cbindgen

### Connection Model

GLIDE uses a **single multiplexed connection per node**. All requests from the application pipeline through this connection, leveraging Valkey's built-in pipelining. This eliminates connection pool management overhead.

When you need isolation (blocking commands like BLPOP, WATCH/UNWATCH, or large value transfers), create a separate GLIDE client instance.

**Lazy connections**: By default, GLIDE defers connection establishment until the first command is executed, reducing startup latency.

### Cluster Topology Management

In cluster mode, GLIDE:
1. Connects to seed nodes provided in configuration
2. Discovers the full cluster topology automatically
3. Uses consensus-based resolution (queries multiple nodes, picks the view with highest agreement)
4. Runs proactive background monitoring for topology changes
5. Throttles refresh requests to prevent server overload
6. Automatically redirects commands on MOVED/ASK errors

Multi-slot commands (MGET, MSET, DEL, UNLINK, EXISTS, TOUCH, WATCH) are automatically split into per-slot sub-commands, dispatched to the correct nodes, and reassembled.

## 4. Basic Usage - Connect, Get/Set, Disconnect

### Python (Async)

```python
import asyncio
from glide import (
    GlideClient,
    GlideClientConfiguration,
    NodeAddress,
)

async def main():
    config = GlideClientConfiguration(
        addresses=[NodeAddress("localhost", 6379)],
        request_timeout=5000,
    )
    client = await GlideClient.create(config)

    try:
        await client.set("greeting", "Hello from GLIDE")
        value = await client.get("greeting")
        print(f"Got: {value.decode()}")  # GLIDE returns bytes
    finally:
        await client.close()

asyncio.run(main())
```

## 5. Cluster Mode vs Standalone Mode

### Standalone Mode

For single-node deployments (optionally with replicas). Use `GlideClient` / `GlideClientConfiguration`.

Configuration includes addresses of the primary and all replica nodes. Supports database selection (`database_id`).

### Cluster Mode

For Valkey Cluster deployments. Use `GlideClusterClient` / `GlideClusterClientConfiguration`.

You only need to provide seed node addresses - GLIDE discovers the full topology automatically.

**Python cluster example:**
```python
from glide import (
    GlideClusterClient,
    GlideClusterClientConfiguration,
    NodeAddress,
    ReadFrom,
)

config = GlideClusterClientConfiguration(
    addresses=[
        NodeAddress("node1.example.com", 6379),
        NodeAddress("node2.example.com", 6380),
    ],
    read_from=ReadFrom.PREFER_REPLICA,
)
client = await GlideClusterClient.create(config)

await client.set("key", "value")
value = await client.get("key")
await client.close()
```

**Java cluster example:**
```java
GlideClusterClientConfiguration config = GlideClusterClientConfiguration.builder()
    .address(NodeAddress.builder().host("node1.example.com").port(6379).build())
    .address(NodeAddress.builder().host("node2.example.com").port(6380).build())
    .readFrom(ReadFrom.PREFER_REPLICA)
    .build();

GlideClusterClient client = GlideClusterClient.createClient(config).get();
```

**Node.js cluster example:**
```javascript
import { GlideClusterClient } from "@valkey/valkey-glide";

const client = await GlideClusterClient.createClient({
    addresses: [
        { host: "node1.example.com", port: 6379 },
        { host: "node2.example.com", port: 6380 },
    ],
    readFrom: "preferReplica",
});
```

**Go cluster example:**
```go
cfg := config.NewClusterClientConfiguration().
    WithAddress(&config.NodeAddress{Host: "node1.example.com", Port: 6379}).
    WithAddress(&config.NodeAddress{Host: "node2.example.com", Port: 6380}).
    WithReadFrom(config.ReadFromPreferReplica)

client, err := glide.NewClusterClient(cfg)
```

### Read Strategies

| Strategy | Behavior |
|----------|----------|
| `PRIMARY` | All reads go to the primary node (default) |
| `PREFER_REPLICA` | Round-robin across replicas, fallback to primary |
| `AZ_AFFINITY` | Prefer replicas in the same availability zone as the client |
| `AZ_AFFINITY_REPLICAS_AND_PRIMARY` | Prefer local AZ replicas, then local AZ primary, then remote |

AZ Affinity requires Valkey 8.0+ and configuration of `availability-zone` on each server node.

## 6. Advanced Features

### 6.1 Batching - Pipelines and Transactions (GLIDE 2.0+)

GLIDE 2.0 unified the Transaction and Pipeline APIs into a single `Batch` / `ClusterBatch` class with an `isAtomic` flag.

**Atomic Batch (Transaction)** - executes as MULTI/EXEC, no interleaving. In cluster mode, all keys must map to the same hash slot.

**Non-Atomic Batch (Pipeline)** - sends multiple commands without atomic guarantees. Can span multiple slots and nodes in cluster mode.

#### Python

```python
from glide import Batch, ClusterBatch

# Atomic transaction (standalone)
tx = Batch(is_atomic=True)
tx.set("account:src", "100")
tx.set("account:dst", "0")
tx.incrby("account:dst", 50)
tx.decrby("account:src", 50)
result = await client.exec(tx, raise_on_error=True)
# ['OK', 'OK', 50, 50]

# Non-atomic pipeline (cluster)
pipe = ClusterBatch(is_atomic=False)
pipe.set("user:1:name", "Alice")
pipe.set("user:2:name", "Bob")
pipe.get("user:1:name")
pipe.get("user:2:name")
result = await client.exec(pipe, raise_on_error=False)
# ['OK', 'OK', b'Alice', b'Bob']
```

#### Java

```java
import glide.api.models.commands.batch.Transaction;

Transaction tx = new Transaction();
tx.set("key", "value");
tx.incr("counter");
tx.get("key");
Object[] result = client.exec(tx).get();
```

#### Node.js

```javascript
import { Transaction } from "@valkey/valkey-glide";

const tx = new Transaction()
    .set("key", "value")
    .get("key");
const result = await client.exec(tx);
// ['OK', 'value']
```

#### Go

```go
import "github.com/valkey-io/valkey-glide/go/v2/pipeline"

// Atomic transaction
tx := pipeline.NewStandaloneBatch(true)
tx.Set("key", "value")
tx.Incr("counter")
tx.Get("key")
results, err := client.Exec(ctx, *tx, true)

// Non-atomic pipeline
pipe := pipeline.NewClusterBatch(false)
pipe.Set("k1", "v1")
pipe.Set("k2", "v2")
pipe.Get("k1")
results, err := client.Exec(ctx, *pipe, false)
```

**Performance impact of batching**: In benchmarks, pipelined approaches showed 190-257% higher throughput compared to individual async requests.

### 6.2 Pub/Sub

GLIDE supports three subscription modes:

| Mode | Command | Description |
|------|---------|-------------|
| Exact | SUBSCRIBE | Subscribe to specific channel names |
| Pattern | PSUBSCRIBE | Subscribe using glob patterns (e.g., `news.*`) |
| Sharded | SSUBSCRIBE | Cluster-mode slot-scoped channels (Valkey 7.0+) |

**Subscription model**: Historically, subscriptions were configured at client creation time (immutable). Starting with GLIDE 2.3, dynamic subscriptions are supported via runtime `subscribe()` / `unsubscribe()` methods.

**Message reception** (Python):
```python
# Blocking - waits for a message
msg = await subscriber.get_pubsub_message()

# Non-blocking - returns None if no message available
msg = subscriber.try_get_pubsub_message()

# Callback-driven (set during client configuration)
def on_message(msg, context):
    print(f"Channel: {msg.channel}, Message: {msg.message}")
```

**Automatic reconnection**: GLIDE automatically resubscribes to all configured channels on reconnection, with exponential backoff.

**Important**: A subscriber client is dedicated to listening - use separate clients for publishing and subscribing.

### 6.3 Lua Scripting

GLIDE provides a `Script` class that handles script caching automatically. On first execution, the script is loaded via SCRIPT LOAD, and subsequent calls use EVALSHA with the cached SHA1 hash.

**Python:**
```python
from glide import Script

script = Script("return {KEYS[1], ARGV[1]}")
result = await client.invoke_script(script, keys=["mykey"], args=["myvalue"])
```

**Java:**
```java
Script luaScript = new Script("return {KEYS[1], ARGV[1]}", false);
ScriptOptions options = ScriptOptions.builder()
    .key("mykey")
    .arg("myvalue")
    .build();
Object[] result = (Object[]) client.invokeScript(luaScript, options).get();
```

**Node.js:**
```javascript
import { Script } from "@valkey/valkey-glide";

const script = new Script("return { KEYS[1], ARGV[1] }");
const result = await client.invokeScript(script, {
    keys: ["mykey"],
    args: ["myvalue"],
});
```

**Go:**
```go
script := options.NewScript(`
    local key = KEYS[1]
    local value = ARGV[1]
    redis.call('SET', key, value)
    return redis.call('GET', key)
`)
result, err := client.InvokeScriptWithOptions(ctx, script, options.ScriptOptions{
    Keys: []string{"mykey"},
    Args: []string{"myvalue"},
})
```

### 6.4 OpenTelemetry (GLIDE 2.0+)

GLIDE emits traces and metrics automatically once an OTel collector endpoint is configured. No code changes required beyond initial setup.

**What you get:**
- A trace span per Valkey command
- A nested `send_command` child span measuring server communication time
- Metrics: request timeouts, retry counts, MOVED errors
- Status fields (OK/Error) on each span

**Configuration (Python):**
```python
from glide import GlideClientConfiguration, OpenTelemetryConfig

otel_config = OpenTelemetryConfig(
    traces_endpoint="http://localhost:4317",
    metrics_endpoint="http://localhost:4317",
    sample_percentage=5,  # 1-5% recommended for production
)
config = GlideClientConfiguration(
    addresses=[NodeAddress("localhost", 6379)],
    open_telemetry=otel_config,
)
```

**Spring Boot:**
```properties
spring.data.valkey.valkey-glide.open-telemetry.enabled=true
spring.data.valkey.valkey-glide.open-telemetry.traces-endpoint=http://localhost:4317
spring.data.valkey.valkey-glide.open-telemetry.metrics-endpoint=http://localhost:4317
```

**Limitations**: SCAN family commands and Lua scripting commands are not yet included in tracing.

### 6.5 AZ Affinity Routing

For cloud deployments with replicas across availability zones, AZ Affinity routes reads to the closest replica, reducing latency by ~500us and avoiding cross-AZ data transfer charges.

**Setup:**

1. Configure each Valkey node with its AZ:
```python
await client.config_set(
    {"availability-zone": "us-east-1a"},
    route=ByAddressRoute(host="node1.example.com", port=6379)
)
```

2. Configure the GLIDE client:
```python
config = GlideClusterClientConfiguration(
    addresses=[NodeAddress("node1.example.com", 6379)],
    read_from=ReadFrom.AZ_AFFINITY,
    client_az="us-east-1a",
)
```

**Cost example**: A two-shard cluster processing 250MB/s of reads can reduce monthly cross-AZ costs from $4,373 to $1,088 with AZ affinity.

### 6.6 TLS and Authentication

**Basic TLS:**
```python
config = GlideClientConfiguration(
    addresses=[NodeAddress("valkey.example.com", 6380)],

```

**TLS with custom certificates:**
```python
from glide import TlsAdvancedConfiguration, AdvancedGlideClientConfiguration

with open("/path/to/ca.pem", "rb") as f:
    ca_cert = f.read()

advanced = AdvancedGlideClientConfiguration(
    tls=TlsAdvancedConfiguration(root_pem_cacerts=ca_cert)

config = GlideClientConfiguration(
    addresses=[NodeAddress("valkey.example.com", 6380)],
    use_tls=True,
    advanced_configuration=advanced,
```

Note: Always read certificate files in binary mode (`"rb"`). The hostname in `NodeAddress` must match the server certificate CN/SAN.

**Authentication:**
```python
from glide import ServerCredentials

config = GlideClientConfiguration(
    addresses=[NodeAddress("localhost", 6379)],
    credentials=ServerCredentials(username="myuser", password="mypass"),

```

## 7. Error Handling

### Error Types

| Error | Description |
|-------|-------------|
| `RequestError` | Base class for all request failures |
| `TimeoutError` | Request timed out (exceeds `request_timeout`) |
| `ConnectionError` | Connection lost; client will attempt reconnection |
| `ExecAbortError` | Transaction was aborted (e.g., WATCH key changed) |
| `ConfigurationError` | Invalid client configuration |

### Python

```python
from glide import (
    GlideClient,
    TimeoutError as GlideTimeoutError,
    ConnectionError as GlideConnectionError,
    RequestError,
)

try:
    value = await client.get("key")
except GlideTimeoutError:
    print("Request timed out - consider increasing request_timeout")
except GlideConnectionError:
    print("Connection lost - client is reconnecting")
except RequestError as e:
    print(f"Request failed: {e}")
```

### Java

```java
try {
    String value = client.get("key").get();
} catch (ExecutionException e) {
    if (e.getCause() instanceof RequestException) {
        System.err.println("Request failed: " + e.getCause().getMessage());
    }
}
```

### Go

```go
val, err := client.Get(ctx, "key")
if err != nil {
    // Check error type
    var connErr *errors.ConnectionError
    if errors.As(err, &connErr) {
        fmt.Println("Connection error - client is reconnecting")
    }
    return
}
if val.IsNil() {
    fmt.Println("Key does not exist")
}
```

### Batch Error Handling

The `raise_on_error` parameter controls behavior:
- `true` - throws/raises on the first error encountered
- `false` - returns errors inline in the response array as `RequestException` objects

```python
result = await client.exec(batch, raise_on_error=False)
for item in result:
    if isinstance(item, RequestError):
        print(f"Command failed: {item}")
    else:
        print(f"Success: {item}")
```

## 8. Best Practices

### Connection Management

- **Do not create connection pools.** GLIDE multiplexes all requests over a single connection per node. Creating multiple clients is only needed for blocking commands or WATCH isolation.
- **Always close clients** when done to clean up resources.
- **Use lazy connections** (default) for reduced startup latency.

### Timeout Configuration

- Default request timeout is 250ms. Adjust based on your workload:
  - Cache lookups: 250-500ms
  - Complex operations (SORT, SCAN): 1000-5000ms
  - Lua scripts: 5000-30000ms
- The timeout covers sending, waiting, and any reconnection attempts.

### Reconnection Strategy

GLIDE uses exponential backoff with jitter by default:

```
delay = rand(0 .. factor * (exponentBase ^ attempts))
```

Once max delay is reached, it stays at that value until reconnection succeeds.

```python
from glide import BackoffStrategy

config = GlideClientConfiguration(
    addresses=[NodeAddress("localhost", 6379)],
    connection_backoff=BackoffStrategy(
        num_of_retries=5,
        factor=100,         # milliseconds
        exponent_base=2,
    ),
)
```

### Cluster Mode Tips

- Provide multiple seed nodes for redundancy during initial connection
- Use hash tags `{tag}` to co-locate related keys on the same slot
- Use AZ Affinity for read-heavy workloads to minimize latency
- Prefer non-atomic batches (pipelines) for cross-slot operations
- Atomic batches (transactions) require all keys in the same hash slot

### Inflight Request Limiting

GLIDE caps concurrent requests at 1000 per connection (based on Little's Law: 50k req/s * 1ms avg response = 50 inflight, with headroom). Excess requests are rejected immediately. If you hit this limit, consider:
- Using batching to combine multiple commands
- Creating additional client instances for parallel connections
- Reviewing whether your workload genuinely needs that concurrency

### OpenTelemetry in Production

- Set sampling to 1-5% to balance observability vs overhead
- Use gRPC or HTTP exporters to an OTel Collector
- File-based export is useful for development/debugging only

| valkey-glide atomic | 3,458 | 0.289ms |
| ioredis pipeline | 3,334 | 0.300ms |

**Key takeaway**: For sequential operations, native clients are ~15-20% faster due to the FFI overhead of GLIDE's Rust core bridge. For batched/transaction workloads, GLIDE is competitive. The performance gap narrows significantly under real-world conditions with cluster topology changes, reconnections, and error handling - where GLIDE's reliability features provide the most value.

### When to Choose GLIDE

Choose GLIDE when you need:
- **Multi-language consistency** - same behavior across Python, Java, Node, Go
- **Production reliability** - automatic failover, reconnection, topology tracking
- **Cloud-optimized deployments** - AZ affinity for cost and latency reduction
- **Observability** - built-in OpenTelemetry without additional instrumentation
- **Enterprise features** - unified client maintained by Valkey core team

Stick with existing clients when:
- Maximum single-language raw throughput is critical
- Your application is simple (single node, no cluster)
- You have deep expertise with your current client
- You need minimal dependencies (GLIDE bundles a Rust binary)

## 10. Migration Guides

### From redis-py to GLIDE (Python)

**Key differences:**
- GLIDE is async-first (sync API available from v2.1)
- Returns bytes, not strings (no `decode_responses=True` equivalent)
- Multiple arguments require lists, not varargs
- Expiry uses `ExpirySet` objects, not keyword arguments
- Conditional operations use enums, not dedicated methods

```python
# redis-py
r = redis.Redis(host='localhost', port=6379, decode_responses=True)
r.set('key', 'value', ex=60)
r.delete('key1', 'key2')

# GLIDE
from glide import ExpiryType, ExpirySet
client = await GlideClient.create(config)
await client.set('key', 'value', expiry=ExpirySet(ExpiryType.SEC, 60))
await client.delete(['key1', 'key2'])
```

### From ioredis to GLIDE (Node.js)

**Key differences:**
- Hash operations use objects instead of spread pairs
- List/set operations require arrays
- Sorted sets use `{score, member}` objects
- SETEX uses options within `set()`
- Script caching is automatic

```javascript
// ioredis
redis.hset('hash', 'k1', 'v1', 'k2', 'v2');
redis.zadd('zset', 1, 'one', 2, 'two');
redis.setex('key', 5, 'value');

// GLIDE
await client.hset('hash', { k1: 'v1', k2: 'v2' });
await client.zadd('zset', [{ score: 1, member: 'one' }, { score: 2, member: 'two' }]);
await client.set('key', 'value', { expiry: { type: TimeUnit.Seconds, count: 5 } });
```

### From Jedis to GLIDE (Java)

**Key differences:**
- All commands return `CompletableFuture<T>`, call `.get()` for synchronous behavior
- Uses builder pattern for configuration
- Multiple arguments use arrays instead of varargs
- SETEX/SETNX use `SetOptions` builder

```java
// Jedis
jedis.set("key", "value");
String val = jedis.get("key");
jedis.setex("key", 5, "value");

// GLIDE
client.set("key", "value").get();
String val = client.get("key").get();
client.set("key", "value",
    SetOptions.builder().expiry(Seconds(5L)).build()).get();
```

**Jedis Compatibility Layer**: For zero-code-change migration, use the drop-in Jedis compatibility wrapper:

```xml
<dependency>
    <groupId>io.valkey</groupId>
    <artifactId>valkey-glide-jedis-compatibility</artifactId>
    <version>2.1.0</version>
</dependency>
```

This implements the Jedis API backed by GLIDE. Not all features are supported - for production, a full migration to native GLIDE API is recommended.

### From go-redis to GLIDE (Go)

**Key differences:**
- Results use `models.Result[T]` with `.Value()` and `.IsNil()` methods
- Multi-argument commands require slices
- Configuration uses structured objects
- Transactions use the batch pipeline API
- Error handling separates nil-check from error-check

```go
// go-redis
err := rdb.Set(ctx, "key", "value", 0).Err()
val, err := rdb.Get(ctx, "key").Result()
if err == redis.Nil { /* not found */ }

// GLIDE
_, err := client.Set(ctx, "key", "value")
val, err := client.Get(ctx, "key")
if val.IsNil() { /* not found */ }
fmt.Println(val.Value())
```

## 11. Spring Data Valkey Integration

For Spring Boot applications, the `spring-data-valkey` project provides auto-configuration with GLIDE as a driver:

```xml
<dependency>
    <groupId>io.valkey.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-valkey</artifactId>
    <version>0.2.0</version>
</dependency>
```

Supports GLIDE, Lettuce, and Jedis drivers. Includes:
- Auto-configuration for connections, templates, repositories, and caching
- Spring Boot Actuator health indicators
- OpenTelemetry integration via properties
